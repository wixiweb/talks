Git Flow

Quand j'ai découvert ça …

  • master = "production ready"
  • Créer un tag pour chaque MEP
  • Git hooks, serveur d'intégration continue, etc.
  • develop vers develop
  • Convention : feature/xyz
  • Tout ce qui fait plus d'un commit
  • Principalement en local
  • Sur le serveur si travail à plusieurs sur une même feature
  • Depuis develop
  • Convention : release/[numéro tag]
  • Première chose à faire : définir la version de la release
  • Corrections mineures liées à la release en elle-même
  • Merge dans master
  • Création d'un tag
  • Merge dans develop
  • Depuis master vers master
  • Convention : hotfix/xyz
  • Correction(s) sur mastrer
  • Création d'un tag
  • Merge dans develop
  • Si release en cours, back-merge dedans au lieu de develop
  • Le travail sur develop n'est pas impacté

Gitlab

Pourquoi Gitlab ?

  • GitHub en version Open Source
  • Maintenu par Gitlab Inc.
  • De plus en plus d'utilisateurs et d'entreprises
  • Mises à jour mensuelles
  • Installation et maintenance faciles
  • Full featured

Gitlab

Fonctionnalités principales

  • Activity feed, network, graphs
  • Repo/file browser, file edition
  • Wiki (+ markdown)
  • Merge requests
  • Issue tracker
  • Code snippets
  • Web Hooks
  • Groupes de projets
  • Gestion de droits
  • Branches privées
  • 2-factor authentification
  • Gitlab Continuous Integration

Repository File Browser

  • Télécharger les sources
  • Édition de fichiers
  • Voir toutes les branches
  • Comparer des branches/commits

Network Graph

  • Plan du dépôt
  • Historique complet

Fill Diff

  • Code review
  • Discussion globale ou sur une ligne précise
  • Deux modes de vue

Merge request

  • Accetper le merge
